# 技能建立器

此技能提供建立有效技能的指南。

## 關於技能

技能是模組化、獨立的套件，透過提供專業知識、工作流程和工具來擴展 Claude 的能力。可以將它們視為特定領域或任務的「入職指南」——它們將 Claude 從通用代理轉變為配備程序性知識的專業代理，而這些知識是任何模型都無法完全具備的。

### 技能提供的功能

1. 專業工作流程 - 特定領域的多步驟程序
2. 工具整合 - 處理特定檔案格式或 API 的說明
3. 領域專業知識 - 公司特定知識、結構描述、商業邏輯
4. 捆綁資源 - 用於複雜和重複性任務的腳本、參考資料和素材

## 核心原則

### 簡潔是關鍵

上下文視窗是公共資源。技能與 Claude 需要的所有其他內容共享上下文視窗：系統提示、對話歷史、其他技能的元資料以及實際的使用者請求。

**預設假設：Claude 已經非常聰明。** 只添加 Claude 尚未具備的上下文。質疑每一條資訊：「Claude 真的需要這個解釋嗎？」以及「這段話值得消耗這些 token 嗎？」

偏好簡潔的範例而非冗長的解釋。

### 設定適當的自由度

根據任務的脆弱性和變異性來匹配具體程度：

**高自由度（基於文字的指示）**：當多種方法都有效、決策取決於上下文或啟發式方法指導方向時使用。

**中自由度（帶參數的虛擬碼或腳本）**：當存在偏好的模式、可接受一些變化或配置影響行為時使用。

**低自由度（特定腳本，少量參數）**：當操作脆弱且容易出錯、一致性至關重要或必須遵循特定順序時使用。

將 Claude 想像成在探索一條路徑：懸崖邊的窄橋需要特定的護欄（低自由度），而開闊的田野允許多條路線（高自由度）。

### 技能的結構

每個技能由必需的 SKILL.md 檔案和可選的捆綁資源組成：

```
skill-name/
├── SKILL.md（必需）
│   ├── YAML frontmatter 元資料（必需）
│   │   ├── name:（必需）
│   │   └── description:（必需）
│   └── Markdown 說明（必需）
└── 捆綁資源（可選）
    ├── scripts/          - 可執行程式碼（Python/Bash 等）
    ├── references/       - 根據需要載入上下文的文件
    └── assets/           - 用於輸出的檔案（範本、圖示、字型等）
```

#### SKILL.md（必需）

每個 SKILL.md 包含：

- **Frontmatter**（YAML）：包含 `name` 和 `description` 欄位。這是 Claude 用來判斷何時使用技能的唯一欄位，因此清楚且全面地描述技能是什麼以及何時使用它非常重要。
- **內文**（Markdown）：使用技能的說明和指導。僅在技能觸發後才載入（如果需要的話）。

#### 捆綁資源（可選）

##### 腳本（`scripts/`）

用於需要確定性可靠性或重複編寫的任務的可執行程式碼（Python/Bash 等）。

- **何時包含**：當相同的程式碼被重複編寫或需要確定性可靠性時
- **範例**：用於 PDF 旋轉任務的 `scripts/rotate_pdf.py`
- **優點**：節省 token、確定性、可以在不載入上下文的情況下執行
- **注意**：腳本可能仍需要被 Claude 讀取以進行修補或環境特定的調整

##### 參考資料（`references/`）

旨在根據需要載入上下文以提供 Claude 處理和思考資訊的文件和參考資料。

- **何時包含**：用於 Claude 在工作時應參考的文件
- **範例**：用於財務結構描述的 `references/finance.md`、用於公司保密協議範本的 `references/mnda.md`、用於公司政策的 `references/policies.md`、用於 API 規格的 `references/api_docs.md`
- **使用情境**：資料庫結構描述、API 文件、領域知識、公司政策、詳細的工作流程指南
- **優點**：保持 SKILL.md 精簡，僅在 Claude 判斷需要時才載入
- **最佳實踐**：如果檔案較大（>10k 字），在 SKILL.md 中包含 grep 搜尋模式
- **避免重複**：資訊應該只存在於 SKILL.md 或參考檔案中，而不是兩者都有。除非資訊確實是技能的核心，否則偏好將詳細資訊放在參考檔案中——這使 SKILL.md 保持精簡，同時使資訊可被發現而不會佔用上下文視窗。在 SKILL.md 中只保留必要的程序性說明和工作流程指導；將詳細的參考資料、結構描述和範例移至參考檔案。

##### 素材（`assets/`）

不打算載入上下文，而是用於 Claude 產生的輸出中的檔案。

- **何時包含**：當技能需要將在最終輸出中使用的檔案時
- **範例**：用於品牌素材的 `assets/logo.png`、用於 PowerPoint 範本的 `assets/slides.pptx`、用於 HTML/React 樣板的 `assets/frontend-template/`、用於字型的 `assets/font.ttf`
- **使用情境**：範本、圖片、圖示、樣板程式碼、字型、被複製或修改的範例文件
- **優點**：將輸出資源與文件分開，使 Claude 能夠使用檔案而不必將它們載入上下文

##### references/ 與 assets/ 的分配規則

**核心區別**：

| 類型 | 目的 | Claude 如何使用 |
|------|------|-----------------|
| `references/` | 提供知識和規格 | 讀取後理解並應用 |
| `assets/` | 提供可複製的模板 | 複製後填充變數 |

**分配決策指南**：

問自己：「Claude 需要**理解**這個檔案，還是需要**複製**這個檔案？」

- 需要理解 → `references/`
- 需要複製 → `assets/`

**命名規則**：

`references/` 檔案：
- 使用描述內容性質的名稱
- 範例：`schema.md`（資料結構）、`api-docs.md`（API 文件）、`policies.md`（政策規範）
- 對於多領域技能，按領域命名：`finance.md`、`sales.md`、`marketing.md`

`assets/` 檔案：
- 使用描述用途的名稱
- 範例：`html-templates.md`（HTML 模板集合）、`email-template.html`（單一郵件模板）
- 對於多種輸出格式，按格式命名：`report-template.docx`、`slides-template.pptx`

**實際範例**：

```
book-notes/
├── SKILL.md                      # 核心工作流程（精簡）
├── references/
│   └── schema.md                 # JSON 資料結構規格
└── assets/
    └── html-templates.md         # HTML 輸出模板
```

在此範例中：
- `schema.md` 放在 `references/` 因為 Claude 需要**理解** JSON 結構才能正確讀寫資料
- `html-templates.md` 放在 `assets/` 因為 Claude 需要**複製**模板並填充變數來生成 HTML

#### 技能中不應包含的內容

技能應只包含直接支援其功能的必要檔案。不要建立多餘的文件或輔助檔案，包括：

- README.md
- INSTALLATION_GUIDE.md
- QUICK_REFERENCE.md
- CHANGELOG.md
- 等等

技能應只包含 AI 代理完成手邊工作所需的資訊。它不應包含關於建立過程的輔助上下文、設定和測試程序、面向使用者的文件等。建立額外的文件檔案只會增加混亂和困惑。

### 漸進式揭露設計原則

技能使用三層載入系統來有效管理上下文：

1. **元資料（name + description）** - 始終在上下文中（約 100 字）
2. **SKILL.md 內文** - 當技能觸發時（<5k 字）
3. **捆綁資源** - 根據 Claude 的需要（無限制，因為腳本可以在不讀入上下文視窗的情況下執行）

#### 漸進式揭露模式

將 SKILL.md 內文保持在必要內容，並控制在 500 行以下以減少上下文膨脹。當接近此限制時，將內容拆分到單獨的檔案中。當將內容拆分到其他檔案時，從 SKILL.md 引用它們並清楚描述何時讀取它們非常重要，以確保技能的讀者知道它們的存在以及何時使用它們。

**關鍵原則：** 當技能支援多種變體、框架或選項時，在 SKILL.md 中只保留核心工作流程和選擇指導。將特定變體的詳細資訊（模式、範例、配置）移至單獨的參考檔案。

**模式 1：帶參考資料的高階指南**

```markdown
# PDF 處理

## 快速開始

使用 pdfplumber 提取文字：
[程式碼範例]

## 進階功能

- **表單填寫**：完整指南請見 [FORMS.md](FORMS.md)
- **API 參考**：所有方法請見 [REFERENCE.md](REFERENCE.md)
- **範例**：常見模式請見 [EXAMPLES.md](EXAMPLES.md)
```

Claude 僅在需要時才載入 FORMS.md、REFERENCE.md 或 EXAMPLES.md。

**模式 2：特定領域的組織**

對於具有多個領域的技能，按領域組織內容以避免載入不相關的上下文：

```
bigquery-skill/
├── SKILL.md（概述和導航）
└── reference/
    ├── finance.md（營收、帳單指標）
    ├── sales.md（商機、銷售管道）
    ├── product.md（API 使用、功能）
    └── marketing.md（活動、歸因）
```

當使用者詢問銷售指標時，Claude 只讀取 sales.md。

同樣，對於支援多個框架或變體的技能，按變體組織：

```
cloud-deploy/
├── SKILL.md（工作流程 + 供應商選擇）
└── references/
    ├── aws.md（AWS 部署模式）
    ├── gcp.md（GCP 部署模式）
    └── azure.md（Azure 部署模式）
```

當使用者選擇 AWS 時，Claude 只讀取 aws.md。

**模式 3：條件式詳細資訊**

顯示基本內容，連結到進階內容：

```markdown
# DOCX 處理

## 建立文件

使用 docx-js 建立新文件。請見 [DOCX-JS.md](DOCX-JS.md)。

## 編輯文件

對於簡單編輯，直接修改 XML。

**追蹤變更**：請見 [REDLINING.md](REDLINING.md)
**OOXML 詳細資訊**：請見 [OOXML.md](OOXML.md)
```

Claude 僅在使用者需要這些功能時才讀取 REDLINING.md 或 OOXML.md。

**重要指南：**

- **避免深層巢狀參考** - 從 SKILL.md 保持一層深度的參考。所有參考檔案應直接從 SKILL.md 連結。
- **為較長的參考檔案建立結構** - 對於超過 100 行的檔案，在頂部包含目錄，以便 Claude 在預覽時可以看到完整範圍。

## 技能建立流程

技能建立包含以下步驟：

1. 透過具體範例理解技能
2. 規劃可重複使用的技能內容（腳本、參考資料、素材）
3. 初始化技能（執行 init_skill.py）
4. 編輯技能（實作資源並撰寫 SKILL.md）
5. 打包技能（執行 package_skill.py）
6. 根據實際使用情況迭代

按順序遵循這些步驟，僅在有明確理由說明它們不適用時才跳過。

### 步驟 1：透過具體範例理解技能

僅在技能的使用模式已經清楚理解時才跳過此步驟。即使在處理現有技能時，此步驟仍然有價值。

要建立有效的技能，需要清楚理解技能將如何使用的具體範例。這種理解可以來自直接的使用者範例或經過使用者回饋驗證的生成範例。

例如，在建立圖像編輯器技能時，相關問題包括：

- 「圖像編輯器技能應該支援什麼功能？編輯、旋轉、還有其他嗎？」
- 「你能給一些這個技能會如何使用的範例嗎？」
- 「我可以想像使用者會要求像『移除這張圖片的紅眼』或『旋轉這張圖片』這樣的事情。你還能想像這個技能會以其他方式被使用嗎？」
- 「使用者會說什麼來觸發這個技能？」

為避免讓使用者不知所措，避免在單一訊息中問太多問題。從最重要的問題開始，根據需要跟進以獲得更好的效果。

當對技能應該支援的功能有清楚的認識時，結束此步驟。

### 步驟 2：規劃可重複使用的技能內容

要將具體範例轉化為有效的技能，透過以下方式分析每個範例：

1. 考慮如何從頭開始執行範例
2. 識別在重複執行這些工作流程時什麼腳本、參考資料和素材會有幫助

範例：當建立 `pdf-editor` 技能來處理像「幫我旋轉這個 PDF」這樣的查詢時，分析顯示：

1. 旋轉 PDF 每次都需要重新編寫相同的程式碼
2. 在技能中儲存 `scripts/rotate_pdf.py` 腳本會很有幫助

範例：當設計 `frontend-webapp-builder` 技能來處理像「幫我建立一個待辦事項應用程式」或「幫我建立一個追蹤步數的儀表板」這樣的查詢時，分析顯示：

1. 編寫前端 webapp 每次都需要相同的 HTML/React 樣板
2. 在技能中儲存包含樣板 HTML/React 專案檔案的 `assets/hello-world/` 範本會很有幫助

範例：當建立 `big-query` 技能來處理像「今天有多少使用者登入？」這樣的查詢時，分析顯示：

1. 查詢 BigQuery 每次都需要重新發現表格結構描述和關係
2. 在技能中儲存記錄表格結構描述的 `references/schema.md` 檔案會很有幫助

要確定技能的內容，分析每個具體範例以建立要包含的可重複使用資源清單：腳本、參考資料和素材。

### 步驟 3：初始化技能

此時，是時候實際建立技能了。

僅在正在開發的技能已經存在，且需要迭代或打包時才跳過此步驟。在這種情況下，繼續下一步。

從頭開始建立新技能時，始終執行 `init_skill.py` 腳本。該腳本方便地生成一個新的範本技能目錄，自動包含技能所需的一切，使技能建立過程更加高效和可靠。

用法：

```bash
scripts/init_skill.py <skill-name> --path <output-directory>
```

腳本會：

- 在指定路徑建立技能目錄
- 生成帶有正確 frontmatter 和 TODO 佔位符的 SKILL.md 範本
- 建立範例資源目錄：`scripts/`、`references/` 和 `assets/`
- 在每個目錄中添加可以自訂或刪除的範例檔案

初始化後，根據需要自訂或移除生成的 SKILL.md 和範例檔案。

### 步驟 4：編輯技能

在編輯（新生成或現有的）技能時，請記住技能是為另一個 Claude 實例使用而建立的。包含對 Claude 有益且不明顯的資訊。考慮什麼程序性知識、特定領域的詳細資訊或可重複使用的素材會幫助另一個 Claude 實例更有效地執行這些任務。

#### 學習經過驗證的設計模式

根據你的技能需求參考這些有用的指南：

- **多步驟流程**：請見 references/workflows.md 了解順序工作流程和條件邏輯
- **特定輸出格式或品質標準**：請見 references/output-patterns.md 了解範本和範例模式

這些檔案包含有效技能設計的最佳實踐。

#### 從可重複使用的技能內容開始

要開始實作，從上面識別的可重複使用資源開始：`scripts/`、`references/` 和 `assets/` 檔案。請注意，此步驟可能需要使用者輸入。例如，在實作 `brand-guidelines` 技能時，使用者可能需要提供品牌素材或範本儲存在 `assets/` 中，或文件儲存在 `references/` 中。

添加的腳本必須透過實際執行來測試，以確保沒有錯誤且輸出符合預期。如果有許多類似的腳本，只需測試具有代表性的樣本以確保它們都能運作，同時平衡完成時間。

技能不需要的任何範例檔案和目錄都應該刪除。初始化腳本在 `scripts/`、`references/` 和 `assets/` 中建立範例檔案以展示結構，但大多數技能不需要所有這些。

#### 更新 SKILL.md

**撰寫指南：** 始終使用祈使句/不定式形式。

##### Frontmatter

撰寫帶有 `name` 和 `description` 的 YAML frontmatter：

- `name`：技能名稱
- `description`：這是技能的主要觸發機制，幫助 Claude 理解何時使用該技能。
  - 包含技能的功能和使用它的特定觸發器/上下文。
  - 在此處包含所有「何時使用」資訊 - 而不是在內文中。內文僅在觸發後才載入，因此內文中的「何時使用此技能」部分對 Claude 沒有幫助。
  - `docx` 技能的描述範例：「全面的文件建立、編輯和分析，支援追蹤變更、註解、格式保留和文字提取。當 Claude 需要處理專業文件（.docx 檔案）時使用：(1) 建立新文件、(2) 修改或編輯內容、(3) 處理追蹤變更、(4) 添加註解或任何其他文件任務」

不要在 YAML frontmatter 中包含任何其他欄位。

##### 內文

撰寫使用技能及其捆綁資源的說明。

### 步驟 5：打包技能

技能開發完成後，必須打包成可分發的 .skill 檔案以分享給使用者。打包過程會自動先驗證技能以確保它符合所有要求：

```bash
scripts/package_skill.py <path/to/skill-folder>
```

可選的輸出目錄指定：

```bash
scripts/package_skill.py <path/to/skill-folder> ./dist
```

打包腳本會：

1. 自動**驗證**技能，檢查：

   - YAML frontmatter 格式和必填欄位
   - 技能命名慣例和目錄結構
   - 描述的完整性和品質
   - 檔案組織和資源參考

2. 如果驗證通過則**打包**技能，建立以技能命名的 .skill 檔案（例如 `my-skill.skill`），包含所有檔案並維護適當的目錄結構以供分發。.skill 檔案是帶有 .skill 副檔名的 zip 檔案。

如果驗證失敗，腳本會報告錯誤並退出而不建立套件。修復任何驗證錯誤並再次執行打包命令。

### 步驟 6：迭代

測試技能後，使用者可能會要求改進。這通常發生在使用技能後不久，對技能表現有新鮮的上下文。

**迭代工作流程：**

1. 在真實任務上使用技能
2. 注意困難或低效之處
3. 識別 SKILL.md 或捆綁資源應該如何更新
4. 實作變更並再次測試

## 更新技能

### `/skill-creator update`
更新本機的 skill-creator skill 到最新版本。

**步驟：**
1. 使用 AskUserQuestion 詢問使用者要更新哪個位置：
   - 全域（~/.claude/）：更新所有專案共用的 skills
   - 當前專案（./.claude/）：只更新當前專案的 skills

2. 根據選擇設定目標路徑

3. 執行更新流程：
   ```bash
   rm -rf /tmp/skill_pool
   git clone --depth 1 https://github.com/JakeChang/skill_pool.git /tmp/skill_pool
   ```

4. 複製 skill-creator skill 到目標位置：
   ```bash
   mkdir -p <目標路徑>/skills
   cp -r /tmp/skill_pool/.claude/skills/skill-creator <目標路徑>/skills/
   ```

5. 清理臨時檔案：
   ```bash
   rm -rf /tmp/skill_pool
   ```

6. 顯示更新結果

**輸出範例：**
```
[更新完成] skill-creator 已更新至最新版本

更新位置：~/.claude/skills/skill-creator
```
